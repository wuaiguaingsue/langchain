# LangChain 发布政策

LangChain 生态系统由不同的组件包组成（例如：`langchain-core`、`langchain`、`langchain-community`、`langgraph`、`langserve`、合作伙伴包等）

## 版本控制

### `langchain`、`langchain-core` 和集成包

`langchain`、`langchain-core`、`langchain-text-splitters` 以及集成包（`langchain-openai`、`langchain-anthropic` 等）遵循[语义化版本控制](https://semver.org/)，格式为 0.**Y**.**Z**。这些包正处于快速开发阶段，因此目前使用 0 作为主版本号。

次要版本增加的情况：

- 对任何未标记为 `beta` 的公共接口进行破坏性更改。

补丁版本增加的情况：

- 错误修复
- 新功能
- 对私有接口的任何更改
- 对 `beta` 功能的任何更改

在次要版本之间升级时，用户应查看破坏性更改和弃用功能的列表。

我们会不定期发布**候选版本**。这些版本计划作为稳定版本发布，但我们希望在正式发布前获得社区的反馈。候选版本的版本号格式为 0.**Y**.**Z**rc**N**。例如：0.2.0rc1。如果没有发现问题，候选版本将以相同的版本号发布为稳定版本。如果发现问题，我们将发布一个新的候选版本，并增加 `N` 的值（例如：0.2.0rc2）。

### `langchain-community`

`langchain-community` 当前版本为 `0.2.x`。

次要版本增加的情况：

- 更新所需 `langchain-x` 依赖的主要/次要版本。例如：将所需的 `langchain-core` 版本从 `^0.2.x` 更新到 `0.3.0`。

补丁版本增加的情况：

- 错误修复
- 新功能
- 对私有接口的任何更改
- 对 `beta` 功能的任何更改
- 对集成进行破坏性更改以反映第三方服务的破坏性更改

我们会尽可能避免在补丁版本中进行破坏性更改。
但是，如果外部 API 发生破坏性更改，则相应的 `langchain-community` 集成可能会在补丁版本中出现破坏性更改。

### `langchain-experimental`

`langchain-experimental` 当前版本为 `0.0.x`。所有更改都将伴随补丁版本的增加。

## 发布节奏

我们预计将 `langchain` 和 `langchain-core` 的**次要**版本（例如：从 0.2.x 到 0.3.0）间隔至少 2-3 个月发布，因为这些版本可能包含破坏性更改。

补丁版本会频繁发布，每周最多几次，因为它们包含错误修复和新功能。

## API 稳定性

LLM 应用程序的开发是一个快速发展的领域，我们不断从用户和社区中学习。因此，我们预计 `langchain` 和 `langchain-core` 中的 API 将继续发展，以更好地满足用户的需求。

尽管 `langchain` 和 `langchain-core` 目前处于 1.0 之前的状态，但我们致力于维护这些包的 API 稳定性。

- 对公共 API 的破坏性更改将导致次要版本的增加（第二位数字）
- 任何错误修复或新功能将导致补丁版本的增加（第三位数字）

我们通常会尽量避免不必要的更改，并为被移除的功能提供弃用政策。

### 其他包的稳定性

LangChain 生态系统中其他包的稳定性可能有所不同：

- `langchain-community` 是一个社区维护的包，包含第三方集成。虽然我们尽力审查和测试 `langchain-community` 中的更改，但由于包含许多社区贡献，`langchain-community` 预计会比 `langchain` 和 `langchain-core` 经历更多的破坏性更改。
- 合作伙伴包可能遵循不同的稳定性和版本控制政策，用户应参考这些包的文档以获取更多信息；然而，通常这些包预计是稳定的。

### 什么是“API 稳定性”？

API 稳定性意味着：

- 所有公共 API（此文档中的所有内容）不会在没有提供向后兼容别名的情况下被移动或重命名。
- 如果向这些 API 添加新功能（这很可能发生），它们不会破坏或改变现有方法的含义。换句话说，“稳定”并不（一定）意味着“完整”。
- 如果由于某种原因，声明为稳定的 API 必须被移除或替换，它将被声明为弃用，但至少会在两个次要版本中保留在 API 中。当调用弃用的方法时，将发出警告。

### **标记为内部的 API**

某些 API 以几种方式明确标记为“内部”：

- 某些文档提到内部内容并将其标记为内部。如果文档中提到某些内容是内部的，则可能会发生更改。
- 以前缀下划线（**`_`**）开头的函数、方法和其他对象。这是 Python 标准的指示某些内容是私有的约定；如果任何方法以单个 **`_`** 开头，则它是内部 API。
    - **例外情况：**某些方法以 `_` 为前缀，但不包含实现。这些方法是*为了*由提供实现的子类重写。这些方法通常是 LangChain 的**公共 API**的一部分。

## 弃用政策

我们通常会避免弃用功能，直到有更好的替代方案可用。

当某个功能被弃用时，它将在当前和下一个次要版本的 `langchain` 和 `langchain-core` 中继续工作。之后，该功能将被移除。

由于我们预计次要版本的间隔至少为 2-3 个月，这意味着某个功能可以在弃用后 2-6 个月内被移除。

在某些情况下，如果某个功能没有对包造成问题，我们可能允许其在代码库中保留更长时间，以减少用户的负担。
